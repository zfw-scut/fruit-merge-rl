# Codex 修改记录规则

本目录用于记录 Codex 对项目做出的较大修改。目标是让后续开发者或 agent 能快速理解：为什么改、改了什么、影响哪些文件、如何验证。

## 何时需要新增记录

以下情况需要新增一条修改记录：

- 修改了游戏主逻辑、强化学习环境、训练流程或公共工具。
- 新增、删除或重命名了重要文件、目录、依赖或资源。
- 修复了会影响运行、训练、兼容性的关键问题。
- 做了较大规模的表现层、交互层、工程结构调整。

以下情况通常不需要单独记录：

- 只修正文案、注释、格式或极小范围 typo。
- 临时调试、不会进入长期代码的实验。
- 已经被同一次较大修改记录涵盖的细碎改动。

## 文件命名

文件名采用连续编号、简短修改信息和日期：

```text
01_修改信息_YYYY_MM_DD.md
02_修改信息_YYYY_MM_DD.md
03_修改信息_YYYY_MM_DD.md
```

示例：

```text
01_创建项目文档结构_2026_06_27.md
02_优化游戏渲染表现_2026_06_28.md
```

编号使用两位数字，从 `01` 开始递增。新增记录前先查看本目录已有最大编号，再继续递增。

## 推荐内容结构

每条记录保持简洁，建议包含：

```markdown
# 标题

## 修改原因

说明为什么需要这次修改。

## 主要修改

- 概括主要改动。
- 标明重要行为变化。

## 涉及文件

- `path/to/file.py`: 简述变化。

## 验证方式

记录运行过的命令、检查结果，或说明未验证的原因。

## 备注

记录兼容性、后续待办或风险。
```

## 写作原则

- 记录决策和影响，不粘贴大段日志。
- 优先写清楚用途、原因、行为变化和验证结果。
- 涉及破坏性变更、依赖变化、兼容性处理时必须明确说明。
- 如果修改没有完成或存在风险，要在记录中直接写出来。
