# 优化 DQN 训练性能主链路

## 修改原因

大规模训练时，原训练链路主要瓶颈集中在 headless 物理模拟、重复构图、逐步采样和 replay 常驻内存。为了支持更长时间训练，需要在不改变游戏规则和模型目标的前提下，提高吞吐并降低内存压力。

## 主要修改

- 训练入口新增 `--physics-mode fast30`，使用 30fps、`max_physics_frames=240`、`stable_frames=6`、`space_iterations=8` 的快速物理配置。
- `RolloutCollector` 缓存非终止 transition 的 `next_graph`，下一步直接作为当前图复用，减少重复构图。
- `ReplayBuffer` 支持热内存 + 冷磁盘分层存储，默认大容量训练只保留最近一部分经验常驻内存，旧经验按段写入 `run_dir/replay_cold`。
- 冷 replay 段写入时会对段内共享 `GraphTensor` 做去重，缓解 `graph` 与 `next_graph` 双份保存带来的磁盘和 I/O 压力。
- 新增 `ParallelRolloutCollector`，多个 worker 进程独立推进 headless 环境，主进程集中训练。
- 训练脚本新增 `--async-rollout`，可让下一批并行采样与当前 DQN 参数更新尽量重叠。
- 训练日志和 `metrics.csv` 增加 profiling 字段，包含采集、构图、张量转换、动作选择、环境 step、训练、采样、前向、target、反向、优化器、评估、保存、绘图和 replay 分层状态。
- `GraphBuilder` 缓存特征列索引，节点/边特征构造时直接填充非零字段，减少 Python dict 中间对象。
- 降低训练脚本默认评估、保存、绘图频率，同时保留实时进度心跳和 ETA。

## 涉及文件

- `src/daxigua_rl/scripts/train_dqn.py`: 接入 fast30、并行采样、异步预采样、分层 replay 和 profiling 日志。
- `src/daxigua_rl/training/collector.py`: 增加 next_graph 缓存和采集阶段耗时统计。
- `src/daxigua_rl/training/parallel_collector.py`: 新增多进程 rollout collector。
- `src/daxigua_rl/training/replay_buffer.py`: 实现热内存 + 冷磁盘 replay、冷缓存采样和图对象去重。
- `src/daxigua_rl/training/dqn.py`: 增加 train_step 内部阶段耗时统计。
- `src/daxigua_rl/graph/builder.py`: 优化特征向量构造。
- `src/daxigua_rl/graph/tensor.py`: 减少 GraphBatch 拼接时不必要的张量包装转换。
- `tests/test_graph_batch_training.py`: 增加 next_graph 缓存、分层 replay 和并行 collector 测试。
- `docs/rl/TRAINING_SPEED_OPTIMIZATION_PLAN.md`: 更新训练性能优化状态和推荐启动参数。
- `docs/project_map/PROJECT_FILE_INDEX.md`: 同步新增文件与训练组件职责。

## 验证方式

- `python -m py_compile src/daxigua_rl/scripts/train_dqn.py src/daxigua_rl/training/parallel_collector.py tests/test_graph_batch_training.py`
- `PYTHONPATH=src conda run --no-capture-output -n python-torch python -m unittest discover -s tests`
- 使用 `fast30 + num_envs=2 + async_rollout + hybrid replay` 运行 3 update 短训练烟测，确认训练入口、worker 同步、replay 写入、profiling 日志和最终 checkpoint 正常。

## 备注

- 并行采样时建议让 `--collect-per-update` 为 `--num-envs` 的整数倍，否则单次采样不会用满所有 worker。
- `--async-rollout` 会让 worker 使用间隔同步的稍旧 online model 作为行为策略；DQN 属于 off-policy 算法，这种做法可接受，但如果需要最严格的同步行为，可以不启用该参数。
- 大规模 CUDA 训练时如果内存仍然紧张，优先降低 `--hot-replay-capacity` 和 `--batch-size`。
