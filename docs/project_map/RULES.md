# 项目文件索引维护规则

本目录用于记录项目各文件的作用、分工、依赖关系和可复用组件，帮助后续开发者或 agent 快速判断该读哪个文件、复用哪个类或函数。

## 维护目标

- 快速回答“这个项目有哪些部分”。
- 快速回答“某个功能应该看哪个文件”。
- 快速回答“已有轮子在哪里，是否可以复用”。
- 避免新 agent 每次都从零扫描整个项目。

## 何时更新

以下情况需要更新项目文件索引：

- 新增、删除、重命名源码文件、资源目录或重要文档。
- 某个文件的职责发生变化。
- 新增了可复用类、函数、训练入口或运行入口。
- 项目依赖、启动方式、资源组织方式发生变化。

## 推荐记录内容

文件索引应至少包含：

- 文件或目录路径。
- 当前职责。
- 主要入口、类或函数。
- 依赖关系。
- 是否适合复用，以及复用时的注意事项。

## 写作原则

- 保持摘要化，不把源码重新讲一遍。
- 对重要入口给出准确路径。
- 对已知问题、兼容性和依赖缺口要直接标注。
- 更新源码职责时，同步更新 `PROJECT_FILE_INDEX.md`。

## Agent 阅读建议

新的 agent 进入项目时，建议先读：

1. `docs/README.md`
2. `docs/project_map/PROJECT_FILE_INDEX.md`
3. `docs/codex/` 下最新几条修改记录

之后再按任务进入具体源码文件。
